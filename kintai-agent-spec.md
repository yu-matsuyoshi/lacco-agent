# 工数管理エージェント 仕様書

## 1. プロジェクト概要

### 1.1 プロジェクト名

工数管理エージェント（Kintai Agent）

### 1.2 目的

社内で利用しているOA Lacco APP（工数管理Slack Bot）への入力を自然言語化し、ユーザーの工数入力業務を効率化する。

### 1.3 スコープ

- **対象範囲（In Scope）:**
  - 自然言語から工数入力コマンド（`add`形式）の生成
  - Google Calendar連携による工数推測
  - A2Aプロトコル対応（Supervisorエージェントからの呼び出し）
  - スタンドアロン動作（Streamlit UI）

- **対象外（Out of Scope）:**
  - OA Lacco APPへの直接的なコマンド実行（Slack Bot不使用）
  - OA Lacco APP本体の改修
  - アンビエント機能（自動通知・定期実行）
  - DynamoDB直接アクセス
  - REST APIの新規構築

### 1.4 成果物

- 動作するエージェント（Strands Agents実装）
- Streamlit UI（スタンドアロンテスト用）
- A2Aサーバー（Supervisor統合用）
- 統合ドキュメント
- デモ環境

## 2. 背景と課題

### 2.1 現状の課題

**毎日の工数入力プロセス:**

```
1. Slack上でOA Lacco APPを開く
2. "ls /project" で案件リストを確認
   → 案件IDを探す（例: 12345）
3. "ls /job" で区分リストを確認
   → 区分IDを探す（例: 1=開発）
4. コマンドを手打ち
   → "add 2026/01/27 12345 1 50"
5. 複数案件がある場合は繰り返し
```

**現状のlaccoのコマンド一覧（一部）**

- help ヘルプコマンド
- add <日付> <案件ID> <区分ID> <割合> 指定した日付に工数を追加
- add -r <月> <案件ID> <区分ID> <割合> 月単位でまとめて入力
- add -t <繰り返し日数> <日付> <案件ID> <区分ID> <割合> 開始日から指定された営業日分繰り返し入力
- add -u <繰り返し終了日> <日付> <案件ID> <区分ID> <割合> 日付から指定された終了日までの営業日に繰り返し入力
- cp <コピー元日付> <コピー先日付> 入力済みの日付からコピーして入力
- ls /data/<日付> 日付を指定して入力済み工数を確認
- ls /data/<年月> 月を指定して入力済み工数を確認
- ls /project 案件IDを全て確認
- ls /project/<案件名のglob> 案件名の一部を指定して案件IDを確認
- ls /job 区分IDを確認

**問題点:**

- 時間がかかる（1回あたり2-3分）
- 案件ID・区分IDを毎回調べる必要がある
- タイポによるエラー
- 認知負荷が高い

### 2.2 解決策

**自然言語入力:**

```
入力: "今日はプロジェクトAで開発50%、会議50%"
↓
自動生成:
add 2026/01/27 12345 1 50
add 2026/01/27 12345 3 50
```

**期待効果:**

- ⏱️ 時間削減: 2-3分 → 30秒（80%削減）
- 🧠 認知負荷軽減: ID暗記不要
- ❌ エラー率低減: タイポ防止

## 3. 要件定義

### 3.1 機能要件

#### FR1: 自然言語からのコマンド生成

- **説明:** ユーザーの自然言語入力から、OA Lacco APPの`add`コマンドを生成する
- **入力例:**
  - "今日はプロジェクトAが50%、メンテが50%"
  - "昨日は午前が会議、午後が開発"
  - "先週月曜から金曜まで、プロジェクトAが100%"
- **出力例:**
  ```
  add 2026/01/27 12345 1 50
  add 2026/01/27 67890 2 50
  ```
- **優先度:** 必須（P0）

#### FR2: 案件名の曖昧性解決

- **説明:** ユーザーが入力した案件名（略称、部分名称など）を、正式な案件IDにマッチングする
- **例:**
  - "プロジェクトA" → 12345
  - "プロA" → 12345
  - "メンテ" → 67890
- **手法:**
  1. 部分一致検索
  2. LLMによるファジーマッチング
- **優先度:** 必須（P0）

#### FR3: 区分の推測

- **説明:** 文脈から作業区分を推測する
- **例:**
  - "開発" → 1
  - "MTG" → 3（会議）
  - "障害対応" → 2（保守）
- **優先度:** 必須（P0）

#### FR4: 日付の解釈

- **説明:** 相対的な日付表現を絶対日付に変換
- **例:**
  - "今日" → 2026/01/27
  - "昨日" → 2026/01/26
  - "先週月曜" → 2026/01/20
  - "1/27" → 2026/01/27
- **優先度:** 必須（P0）

#### FR5: 割合の検証

- **説明:** 合計が100%になることを確認
- **動作:**
  - 合計が100%でない場合は警告を表示
  - 不足分の提案（例: "残り20%を社内業務として登録しますか？"）
- **優先度:** 必須（P0）

#### FR6: Google Calendar連携

- **説明:** Google Calendarの予定から工数を推測
- **入力:** 日付（デフォルト: 今日）
- **処理:**
  1. カレンダーイベントを取得
  2. イベントタイトルから案件・区分を推測
  3. イベント時間から割合を計算（稼働時間8時間として）
- **出力:** 工数コマンド + 推測の説明
- **優先度:** 高（P1）

#### FR7: A2Aプロトコル対応

- **説明:** Supervisorエージェントから呼び出し可能なA2Aサーバーとして動作
- **対応アクション:**
  - `generate_from_text`: 自然言語から生成
  - `generate_from_calendar`: カレンダーから生成
  - `validate_commands`: コマンド検証
  - `explain_commands`: コマンド説明
- **優先度:** 必須（P0）

#### FR8: スタンドアロンUI

- **説明:** Supervisorなしでも動作するStreamlit UI
- **用途:** 開発・テスト・デモ
- **優先度:** 高（P1）

### 4.2 処理フロー

#### 4.2.1 自然言語入力フロー

```
User Input: "今日はプロジェクトAが50%、メンテが50%"
    ↓
[1] Strands Agent起動
    ↓
[2] Claude API呼び出し
    - System Prompt: エージェントの役割
    - User Message: ユーザー入力
    - Tools: [get_projects, get_categories]
    ↓
[3] Claude APIが推論
    "案件名が不明確 → マスタデータが必要"
    ↓
[4] Tool Call: get_projects()
    → projects.csvを読み込み
    → Claude APIに返す
    ↓
[5] Claude APIが推論
    "プロジェクトA → ID 12345"
    "メンテ → ID 67890"
    "文脈から開発と保守を推測"
    ↓
[6] Tool Call: get_categories()
    → categories.csvを読み込み
    → Claude APIに返す
    ↓
[7] Claude APIが最終推論
    "開発 → ID 1"
    "保守 → ID 2"
    "日付: 今日 → 2026/01/27"
    "割合: 50%, 50% (合計100% ✓)"
    ↓
[8] コマンド生成
    add 2026/01/27 12345 1 50
    add 2026/01/27 67890 2 50
    ↓
[9] User Output
```

#### 4.2.2 カレンダー連携フロー

```
User Request: "カレンダーから推測"
    ↓
[1] Google Calendar API呼び出し
    - 今日の予定を取得
    ↓
[2] イベント解析
    例:
    - 10:00-12:00 "プロジェクトA定例MTG" → 2時間
    - 13:00-17:00 "開発タイム" → 4時間
    ↓
[3] 各イベントをClaude APIで分析
    - "プロジェクトA定例MTG"
      → 案件: プロジェクトA (12345)
      → 区分: 会議 (3)
      → 割合: 2h/8h = 25%

    - "開発タイム"
      → 案件: プロジェクトA (12345) ※文脈から推測
      → 区分: 開発 (1)
      → 割合: 4h/8h = 50%
    ↓
[4] 不足分の計算
    - 合計: 75%
    - 残り: 25% → "社内業務として登録?"
    ↓
[5] コマンド生成
    add 2026/01/27 12345 3 25
    add 2026/01/27 12345 1 50
    add 2026/01/27 99999 4 25  # 社内業務
    ↓
[6] User Output（説明付き）
```

### 5.2 詳細仕様

#### F001: 自然言語入力

**入力パターン:**

```
パターン1: 基本形
"今日はプロジェクトAが50%、メンテが50%"

パターン2: 時間表現
"午前は会議、午後は開発"

パターン3: 繰り返し
"先週月曜から金曜まで、プロジェクトAが100%"

パターン4: 複雑な表現
"今日はプロジェクトAで開発を午前2時間、午後は会議が3時間、残りは社内業務"
```

**処理ステップ:**

1. 日付を特定
2. 案件名を抽出
3. 区分を抽出または推測
4. 割合を計算
5. 検証（合計100%）
6. コマンド生成

**出力形式:**

```json
{
  "success": true,
  "commands": ["add 2026/01/27 12345 1 50", "add 2026/01/27 67890 2 50"],
  "explanation": "プロジェクトA（開発）50%とメンテナンス（保守）50%で登録します",
  "warnings": [],
  "suggestions": []
}
```

#### F002: 案件マッチング

**マッチングアルゴリズム:**

```python
def match_project(user_input: str) -> dict:
    """
    Step 1: 完全一致
    "プロジェクトA本体開発" == "プロジェクトA本体開発"
    → 即座にマッチ

    Step 2: 部分一致
    "プロジェクトA" in "プロジェクトA本体開発"
    → 候補が1件なら確定
    → 候補が複数ならStep 3へ

    Step 3: LLMで選択
    候補リストをClaude APIに渡して最適なものを選択

    Step 4: LLMで全検索
    候補がない場合、全案件リストから類似度で検索
    （案件数が多い場合は上位50件のみ）
    """
```

**信頼度レベル:**

- `high`: 完全一致または部分一致で1件のみ
- `medium`: LLMで選択（候補から）
- `low`: LLMで全検索（曖昧）

**出力:**

```json
{
  "project_id": 12345,
  "project_name": "プロジェクトA本体開発",
  "confidence": "high",
  "method": "partial_match"
}
```

#### F006: カレンダー連携

**Google Calendar API使用:**

- スコープ: `calendar.readonly`
- 取得範囲: 指定日の0:00〜23:59
- イベント情報:
  - タイトル
  - 開始時刻
  - 終了時刻

**推測ロジック:**

```python
# 1. イベントごとに分析
for event in calendar_events:
    # タイトルから案件を推測
    project = llm_infer_project(event.title)

    # タイトルから区分を推測
    category = llm_infer_category(event.title)

    # 時間から割合を計算
    duration_hours = (event.end - event.start).hours
    percentage = (duration_hours / 8) * 100

    entries.append({
        "project_id": project.id,
        "category_id": category.id,
        "percentage": percentage,
        "reason": f"カレンダー: {event.title} ({duration_hours}時間)"
    })

# 2. 不足分の計算
total = sum(e.percentage for e in entries)
if total < 100:
    unaccounted = 100 - total
    suggestions.append(f"残り{unaccounted}%を社内業務として登録しますか？")
```

## 6. データモデル

### 6.1 案件マスタ（projects.csv）

**ファイル形式:** CSV（UTF-8）

**スキーマ:**

```csv
id,name
```

**例:**

```csv
id,name
12345,プロジェクトA本体開発
67890,メンテナンス業務システム
11111,新規ECサイト構築
22222,既存システム改修
33333,インフラ基盤整備
99999,社内業務
```

**備考:**

- `id`: 案件ID（OA Lacco APPで使用される数値ID）
- `name`: 案件名（正式名称）
- keywordsカラムは不要（LLMで柔軟にマッチング）

### 6.2 区分マスタ（categories.csv）

**ファイル形式:** CSV（UTF-8）

**スキーマ:**

```csv
id,name
```

**例:**

```csv
id,name
1,開発
2,保守
3,会議
4,調査
5,設計
6,テスト
7,レビュー
8,その他
```

**備考:**

- `id`: 区分ID（OA Lacco APPで使用される数値ID）
- `name`: 区分名

## 11. デプロイ

### 11.1 ローカル開発環境

```bash
# 1. リポジトリクローン
git clone https://github.com/your-org/kintai-agent.git
cd kintai-agent

# 2. 仮想環境作成
python -m venv venv
source venv/bin/activate  # Windows: venv\Scripts\activate

# 3. 依存関係インストール
pip install -r requirements.txt

# 4. 環境変数設定
cp .env.example .env
# .envを編集

# 5. Google Calendar API設定
# credentials.jsonを配置

# 6. Streamlit UI起動
streamlit run ui/streamlit_app.py

# 7. A2Aサーバー起動（別ターミナル）
python main.py
```

### 11.2 Docker環境

```dockerfile
# Dockerfile
FROM python:3.11-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

EXPOSE 8000 8501

CMD ["python", "main.py"]
```

```yaml
# docker-compose.yml
version: "3.8"

services:
  kintai-agent:
    build: .
    ports:
      - "8000:8000" # A2A
      - "8501:8501" # Streamlit
    environment:
      - ANTHROPIC_API_KEY=${ANTHROPIC_API_KEY}
    volumes:
      - ./data:/app/data
      - ./credentials.json:/app/credentials.json
      - ./token.pickle:/app/token.pickle
```

```bash
# 起動
docker-compose up
```
