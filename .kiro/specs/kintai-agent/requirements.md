# 要件定義書

## はじめに

工数管理エージェント（Kintai Agent）は、社内で利用しているOA Lacco APP（工数管理Slack Bot）への入力を自然言語化し、ユーザーの工数入力業務を効率化するシステムです。ユーザーは自然言語で工数を入力でき、システムが自動的にOA Lacco APPのコマンド形式に変換します。また、Google Calendarとの連携により、予定から工数を推測する機能も提供します。

## 用語集

- **システム**: 工数管理エージェント（Kintai Agent）
- **ユーザー**: 工数入力を行う社内ユーザー
- **OA Lacco APP**: 既存の工数管理Slack Bot
- **コマンド**: OA Lacco APPで使用される工数入力コマンド（例: `add 2026/01/27 12345 1 50`）
- **案件**: プロジェクト。案件IDと案件名で管理される
- **区分**: 作業区分（開発、保守、会議など）。区分IDと区分名で管理される
- **割合**: 工数の割合（0-100の整数）
- **案件マスタ**: 案件マスタデータ（CSV形式）
- **区分マスタ**: 区分マスタデータ（CSV形式）
- **AgentCore Runtime**: Amazon Bedrock AgentCoreのマネージドエージェントホスティング環境
- **AgentCore Gateway**: 外部サービスやツールとの統合を提供するマネージドサービス
- **MCPサーバー**: Model Context Protocol準拠のツールサーバー
- **Lambda Target**: AgentCore Gatewayに登録されたLambda関数ベースのツール
- **A2Aプロトコル**: Agent-to-Agent通信プロトコル（AgentCore Runtime経由で他のエージェントと通信）
- **カレンダーイベント**: Google Calendarの予定イベント
- **自然言語入力**: ユーザーによる自然言語での工数入力
- **CDK**: AWS Cloud Development Kit（インフラストラクチャをコードで定義）

## 要件

### 要件1

**ユーザーストーリー:** ユーザーとして、自然言語で工数を入力したい。なぜなら、案件IDや区分IDを暗記せずに素早く作業を記録できるからです。

#### 受入基準

1. WHEN ユーザーが工数を説明する自然言語入力を提供する THEN システムは入力を解析し、日付、案件名、区分、割合の情報を抽出する
2. WHEN システムが自然言語入力から案件名を抽出する THEN システムは案件マスタから案件IDにマッチングする
3. WHEN システムが自然言語入力から区分を抽出する THEN システムは区分マスタから適切な区分IDを推測する
4. WHEN システムが自然言語入力から日付表現を抽出する THEN システムは相対日付（例：「今日」「昨日」）をYYYY/MM/DD形式の絶対日付に変換する
5. WHEN システムが自然言語入力からコマンドを生成する THEN システムは有効なOA Lacco APPコマンド形式を出力する

### 要件2

**ユーザーストーリー:** ユーザーとして、曖昧な案件名を解決してほしい。なぜなら、略称や部分名称を使ってもエラーなく入力できるからです。

#### 受入基準

1. WHEN ユーザーが案件マスタのエントリと完全一致する案件名を入力する THEN システムは即座に対応する案件IDを返す
2. WHEN ユーザーが1つの案件マスタエントリと部分一致する案件名を入力する THEN システムは対応する案件IDを返す
3. WHEN ユーザーが複数の案件マスタエントリと部分一致する案件名を入力する THEN システムはLLM推論を使用して最も適切な案件IDを選択する
4. WHEN ユーザーが部分一致しない案件名を入力する THEN システムはLLM推論を使用してすべての案件を検索し、最も類似した案件IDを返す
5. WHEN システムが案件名をマッチングする THEN システムは信頼度レベル（high、medium、low）と使用したマッチング方法を提供する

### 要件3

**ユーザーストーリー:** ユーザーとして、工数の割合が合計100%になることを検証してほしい。なぜなら、正確な時間追跡を保証できるからです。

#### 受入基準

1. WHEN システムがコマンドを生成する THEN システムはすべてのエントリの合計割合を計算する
2. WHEN 合計割合が100%に等しい THEN システムは検証を成功としてマークする
3. WHEN 合計割合が100%未満である THEN システムは警告を提供し、残りの割合をどのように割り当てるかを提案する
4. WHEN 合計割合が100%を超える THEN システムは超過量を示すエラーメッセージを提供する
5. WHEN システムが割合の不一致を検出する THEN システムは修正のための提案を提供する

### 要件4

**ユーザーストーリー:** ユーザーとして、Google Calendarのイベントから工数コマンドを生成したい。なぜなら、予定された会議やタスクに基づいて自動的に時間を追跡できるからです。

#### 受入基準

1. WHEN ユーザーが特定の日付のカレンダーベース生成を要求する THEN システムはGoogle Calendar APIからその日付のすべてのカレンダーイベントを取得する
2. WHEN システムがカレンダーイベントを取得する THEN システムは各イベントのイベントタイトル、開始時刻、終了時刻を抽出する
3. WHEN システムがカレンダーイベントを分析する THEN システムはLLMを使用してイベントタイトルから案件名を推測する
4. WHEN システムがカレンダーイベントを分析する THEN システムはLLMを使用してイベントタイトルから区分を推測する
5. WHEN システムがカレンダーイベントを分析する THEN システムはイベント時間を8時間で割った値に基づいて割合を計算する
6. WHEN カレンダーイベントからの合計割合が100%未満である THEN システムは未計上時間を計算し、社内業務として割り当てることを提案する
7. WHEN システムがカレンダーからコマンドを生成する THEN システムは推測された各案件と区分の説明を提供する

### 要件5

**ユーザーストーリー:** システム管理者として、エージェントとツールをAWS CDKでデプロイしたい。なぜなら、インフラ全体をコードで管理し、再現可能なデプロイを実現できるからです。

#### 受入基準

1. WHEN システムがデプロイされる THEN システムはAWS CDKを使用してインフラストラクチャを定義する
2. WHEN CDKスタックがデプロイされる THEN システムはS3バケット、Lambda関数、AgentCore Gateway、AgentCore Runtimeを作成する
3. WHEN ツールが実装される THEN システムはLambda関数としてツールを実装し、AgentCore Gatewayに登録する
4. WHEN エージェントがツールを使用する THEN システムはAgentCore Gateway経由でLambda関数を呼び出す
5. WHEN 他のエージェントがA2AプロトコルでKintai Agentを呼び出す THEN システムはAgentCore Runtime経由でリクエストを受け入れる
6. WHEN マスタデータが必要な場合 THEN システムはS3バケットからCSVファイルを読み込む
7. WHEN Google Calendar連携が必要な場合 THEN システムはAgentCore Gateway経由でMCPサーバーにアクセスする

### 要件6

**ユーザーストーリー:** ユーザーとして、ローカルのWebインターフェースからエージェントを使用したい。なぜなら、開発環境から簡単にエージェントをテストできるからです。

#### 受入基準

1. WHEN ユーザーがローカルでStreamlit UIを起動する THEN システムは自然言語入力とカレンダー連携タブを持つWebインターフェースを表示する
2. WHEN ユーザーがUIで自然言語入力を入力する THEN システムはboto3を使用してAgentCore Runtime上のエージェントを呼び出し、コマンドを生成して表示する
3. WHEN ユーザーがカレンダー連携タブを選択する THEN システムは日付選択を許可し、カレンダーベース分析を表示する
4. WHEN システムがUIでコマンドを生成する THEN システムはコマンドをクリップボードにコピーするコピーボタンを提供する
5. WHEN システムがUIで結果を表示する THEN システムはマッチした案件、区分、信頼度レベルを含む詳細な推論情報を表示する

### 要件7

**ユーザーストーリー:** ユーザーとして、さまざまな自然言語パターンを処理してほしい。なぜなら、工数をさまざまな方法で表現できるからです。

#### 受入基準

1. WHEN ユーザーが「プロジェクトA 50%、プロジェクトB 50%」のような基本パターンを入力する THEN システムは正しく解析してコマンドを生成する
2. WHEN ユーザーが「午前は会議、午後は開発」のような時間ベースの表現を入力する THEN システムは適切な割合を推測する
3. WHEN ユーザーが「先週月曜から金曜まで」のような日付範囲を入力する THEN システムは範囲内の各営業日のコマンドを生成する
4. WHEN ユーザーが複数の時間単位を含む複雑な表現を入力する THEN システムは8時間の勤務日に基づいて割合を計算する
5. WHEN システムが曖昧な入力に遭遇する THEN システムは明確化を要求するか、説明付きで合理的な推測を行う

### 要件8

**ユーザーストーリー:** 開発者として、明確なエラーメッセージと警告を提供してほしい。なぜなら、入力の問題を理解して修正できるからです。

#### 受入基準

1. WHEN システムが無効な入力に遭遇する THEN システムは問題を説明するエラーメッセージを返す
2. WHEN システムが低信頼度の案件マッチを行う THEN システムはレスポンスに警告を含める
3. WHEN システムがマッチする案件を見つけられない THEN システムは有効な案件名の提案とともにエラーを返す
4. WHEN システムが割合検証の問題を検出する THEN システムは不一致に関する具体的な警告を提供する
5. WHEN システムが入力を正常に処理する THEN システムはエラーなしで成功ステータスを返す

### 要件9

**ユーザーストーリー:** システム管理者として、CSVファイルから案件と区分データをロードしてほしい。なぜなら、マスタデータを簡単に維持・更新できるからです。

#### 受入基準

1. WHEN システムが起動する THEN システムはidとnameカラムを含むCSVファイルから案件マスタデータをロードする
2. WHEN システムが起動する THEN システムはidとnameカラムを含むCSVファイルから区分マスタデータをロードする
3. WHEN システムが案件をマッチングする必要がある THEN システムはロードされた案件マスタデータを検索する
4. WHEN システムが区分を推測する必要がある THEN システムはロードされた区分マスタデータを参照する
5. WHEN システムがマスタデータファイルをロードできない THEN システムは欠落または無効なファイルを示すエラーを返す

### 要件10

**ユーザーストーリー:** ユーザーとして、1つの入力で複数のコマンド生成をサポートしてほしい。なぜなら、1回のやり取りで複数のプロジェクトにわたる作業を記録できるからです。

#### 受入基準

1. WHEN ユーザーが複数の案件の工数を入力する THEN システムは各案件-区分の組み合わせに対して個別のコマンドを生成する
2. WHEN システムが複数のコマンドを生成する THEN システムは各コマンドが正しい形式に従うことを保証する
3. WHEN システムが複数のコマンドを生成する THEN システムはすべてのコマンドの合計割合が100%に等しいことを検証する
4. WHEN システムが複数のコマンドを出力する THEN システムはそれらを明確で順序付けられたリストで提示する
5. WHEN システムが複数のコマンドを生成する THEN システムはすべてのエントリをカバーする統合された説明を提供する
